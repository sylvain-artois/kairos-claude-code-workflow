## API / Endpoints

This service exposes tRPC procedures rather than REST. Documented here are the procedure groups (each maps to one router file under `src/server/routers/`).

| Procedure group | Type | Description |
|---|---|---|
| `analysis.list` | query | Paginated list of analyses for a business |
| `analysis.byId` | query | Fetch one analysis with full breakdown |
| `crawl.trigger` | mutation | Forward a crawl request to `acme_api` |
| `content.drafts` | query | List generated drafts pending review |
| `content.publish` | mutation | Mark a draft as published, write back to `acme_api` |
| `auth.session` | query | Current session info (Clerk-backed) |

REST endpoints are limited to a thin BFF layer:

| Method | Route | Description |
|--------|-------|-------------|
| GET | `/api/health` | Liveness probe used by compose healthcheck |
| POST | `/api/webhooks/clerk` | Clerk user lifecycle webhook |
| POST | `/api/webhooks/stripe` | Stripe subscription events |

## Database Tables

| Table | Access | Description |
|-------|--------|-------------|
| `users` | READ/WRITE | User accounts (synced from Clerk webhook) |
| `business_members` | READ/WRITE | Role-based access between users and businesses |
| `dashboard_settings` | READ/WRITE | Per-user UI preferences |
| `subscriptions` | READ/WRITE | Stripe subscription state |
| `analysis_runs` | READ | Owned by `acme_api`; dashboard reads only |
| `generated_content` | READ | Owned by `acme_api`; publish mutation writes a status field |

## Dependencies

**Upstream** (this service calls):
- `acme_api` (analysis + content APIs)
- PostgreSQL (shared instance)
- Clerk (auth)
- Stripe (billing)
- PostHog (analytics, optional)

**Downstream** (calls this service):
- End-user browsers (the only direct consumer)
- Clerk + Stripe webhook deliverers

## Behavioral Contracts

**WHEN** a tRPC procedure is invoked without a valid Clerk session
**THEN** the procedure returns `UNAUTHORIZED` before any database access.

**WHEN** `crawl.trigger` is called for a business the current user does not belong to (`business_members` lookup)
**THEN** the mutation returns `FORBIDDEN` and no request is forwarded to `acme_api`.

**WHEN** a Clerk webhook arrives with an invalid signature
**THEN** the route returns 400 and logs the event id without modifying the database.

**WHEN** the Next.js server-side renderer cannot reach `acme_api` within 2s
**THEN** the analysis page falls back to a cached snapshot from the previous render and surfaces a "stale data" banner.

## Known Limitations

- No real-time updates: the dashboard polls every 30s for analysis status — there is no websocket / SSE channel yet.
- No offline mode; the dashboard requires both `acme_api` and the database to be reachable.
- Pagination is offset-based on `analysis_runs` and degrades past ~10k rows per business.
- Mobile layout is functional but not the primary target — design QA is desktop-first.
